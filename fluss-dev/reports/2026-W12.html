FIP-17: Streaming KV Scan RPC

3 条回复 · 3 位参与者

讨论主要围绕 FIP-17（Streaming KV Scan RPC）的设计展开，重点探讨了服务端状态管理机制、API 接口统一性以及拉取式分页与推送式流 RPC 的优劣。经过权衡，社区决定保留基于 snapshot 的拉取式模型，并统一了 API 设计规范。

@Lorenzo Affetti: 建议简化服务端状态管理（使用 snapshot_id 替代 ScannerManager），并提议先实现通用的流式 RPC 原语以支持推送模式。

@Jark Wu: 建议复用现有 Table.newScan API 以保持一致性，使用有符号整数简化 Protobuf 定义，并论证了拉取式模型在流控和容错上的优势。

@Giannis Polyzos: 采纳了关于 API 设计和配置命名的建议，已更新 FIP 以遵循现有的 BatchScanner 设计模式。

已达成初步共识，Giannis 已根据反馈更新了 FIP 文档，将沿用现有的 BatchScanner API 设计和拉取式分页机制。

FIP-33: JBOD Phase 1 - Local Multi-Directory Support

0 条回复 · 1 位参与者

Liu han 发起关于 FIP-33 的讨论，旨在为 Fluss 引入 JBOD 支持以充分利用多磁盘服务器的容量和 I/O 带宽。该计划分三阶段实施，本次讨论聚焦第一阶段“本地多目录支持”，目标是在不改变元数据和协调器的情况下，实现 TabletServer 多目录配置及静态 bucket 放置。

@liu han: 当前单目录设计无法充分利用多磁盘服务器的磁盘容量和 I/O 带宽。

@liu han: JBOD 实施分为三阶段，第一阶段专注于本地多目录支持、静态 bucket 放置和重启恢复，保持 log 与 kv 共置。

@liu han: 第一阶段设计保持低风险，不涉及元数据、RPC 或协调器变更，且兼容现有单目录部署模式。

目前处于提案讨论阶段，作者欢迎社区对 FIP-33 提出反馈和建议。
