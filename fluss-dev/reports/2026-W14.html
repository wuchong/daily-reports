Slack Workspace Down

9 条回复 · 8 位参与者

@Jark Wu: Slack 支持团队同意在确认美国账单地址后恢复工作区，并提议 Discord 作为拥有无限历史的备选平台。

@Michael Koepf: 支持新建工作区或迁移 Discord，但强调应统一平台以避免沟通分散和管理困难。

目前等待 Slack 支持团队确认恢复工作区；若恢复失败，社区倾向于创建新工作区或迁移至 Discord。

Proposal: Native Bitmap Integration & Stateless Pushdown Aggregation

5 条回复 · 2 位参与者

讨论围绕 Fluss 的原生 RoaringBitmap 集成提案（FIP-37）展开，旨在引入 BITMAP 类型及相关 SQL 函数以支持服务端聚合下推。提案作者确认了 FIP 编号并创建了文档，社区成员针对 API 兼容性、Null 语义及功能范围提出了详细建议，作者已据此更新了提案内容。

@Giannis Polyzos: 肯定了提案方向，但指出 DataTypeVisitor 接口变更可能破坏兼容性、Null 语义描述存在矛盾，并质疑 BITMAP_AND_AGG 在缺乏服务端支持下的安全性。

@Prajwal Banakar: 回应称将提供 DataTypeVisitor 的默认实现以避免破坏性变更，修正 Null 语义定义（NULL 返回 NULL，空 Bitmap 返回 0），并将 64 位支持列为第二阶段交付内容。

@Prajwal Banakar: 提到已有一个外部 UDF 库 的 PR，可作为原生支持落地前的过渡补充。

提案已根据社区反馈完成更新，解决了兼容性和语义歧义问题，下一步将继续完善细节并推进实施。

Analysis of Lance storage format support

3 条回复 · 2 位参与者

讨论主要围绕 Fluss 对向量数据及 Lance 存储格式的支持方向展开，重点探讨了将 Fluss 定位为“嵌入管道”的架构模式。参与者一致认为应优先关注数据摄入与快速服务能力，而非过早深入向量处理细节，同时建议引入原生 VECTOR 类型以增强语义感知。

@Giannis Polyzos: 提出“Fluss as embedding pipeline”架构模式，整合数据摄入、Flink批处理嵌入及 Lance 分层，简化系统复杂度。

@Giannis Polyzos: 建议引入原生 VECTOR 数据类型，参考 Paimon 提案，使多模态数据的设计意图更加明确。

@Keith Lee: 指出当前多模态支持缺乏类型系统感知，建议探索日志表的“追加后丰富”功能以减少数据移动。

初步确定以“Fluss 作为嵌入管道”为切入点，下一步将整理相关讨论形成文档以便深入探讨，并考虑支持原生 VECTOR 类型及日志表原地更新功能。
